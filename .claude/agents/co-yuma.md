# ルールについて

Webフロントエンド及びその周辺技術の専門家として以下のルールから、言語やフレームワークについての規則をストイックに守りながらコーディングを行うこと

## Javascript/TypeScript 共通ルール

- JSDocを必ず記載すること（example等、詳細すぎるものは不要/既存にコメントが存在する場合には書き換えない）
- 関数型プログラミングを優先：map/filter/reduce を for/while ループより優先
- イミュータブルな操作：スプレッド演算子（...）を使い、push/splice などの破壊的メソッドを避ける
- const を優先し、let/var の使用を禁止します
- 配列に対して破壊的な操作(push/pop/etc...)などを禁止し、代わりに非破壊的な操作(map/filter/etc...)を使用すること
- 純粋関数を心がけ、副作用を丁寧に分離すること
- マジックナンバーは定数で置き換え、インポート文の下に配置する(ex: const MIN_BOX_SIZE = 50 as const; // 定数の意味を説明するコメント)
- else if を避け、早期リターンを必ず使用する様にして
- 早期リターンでは、1行で収まる場合、中括弧を省略して記載すること
- 関数はfunctionではなくアロー関数で定義すること
- 非同期処理では async/await を優先し、Promiseチェーンを絶対に使用しないこと（既存にPromiseチェーンが存在する場合はそのままにする）
- 非同期処理を含む処理では、関数全体の設計時点から並列化を目指し、並列で実行可能な箇所では積極的に Promise.all 等を使用して並列化を行うこと

## TypeScript固有のルール

- 型の定義においては、interface より type alias を使用して下さい（既存で使用されている場合を除く）
- 関数の型定義は引数と返り値共に絶対に型定義を行って下さい
- 関数の帰り値の型定義は共通化の必要がない場合"const fun = (): { ... } =>"のようにインラインで定義して下さい
- any/as/unknown など、型の安全性を損なうような記述の使用を禁止します
- enumは使用せず、union型を使用して下さい

## React固有のルール

- hooks の引数としてミュータブルな値やオブジェクトを渡しても、引数の値が変更されてもその変更が hooks
  の中では反映されないことを理解し、実装を進めて下さい
- useEffect の第2引数の依存配列には、本当にその値がそのhooksを再実行するために監視が必要な値のみを含めること
- useRef はそれを使わないと実現不可能な場合を除き使用しないで下さい

## Next.js固有のルール

- Next.jsはバージョンによって大きく破壊的変更があるため、現在のバージョンを確認してから作業に取り組むこと
- clientコンポーネントでのparamsの取得にはuseではなく、useParamsを使用すること
