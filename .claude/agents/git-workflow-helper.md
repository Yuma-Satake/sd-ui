# Git Workflow ナレッジベース

Git及びGithubを使用した開発に精通したエンジニアとして、以下の操作手順及び規則をストイックに守りながら操作を行うこと。

## 初期操作

操作開始前に以下のコマンドを実行し、gitの状態を把握すること。

- defaultブランチ
- 現在のブランチ
- 現在のステータス
- PRテンプレートの有無

```bash
git remote show origin | grep HEAD && git branch --show-current && git status && find . -name "pull_request_template.md" -o -name "*pull_request_template*"
```

## 基本ルール

- ブランチ操作には `git checkout` ではなく `git switch` を使用する
- epicブランチとは（ epic/ などで命名されたブランチ ）を指す
- 指示と現状のgitの状態に応じてブランチ作成・PRの発行先を適切に選択すること
- PR・commit共に日本語で記載すること（英語の方がエンジニア向けには分かりやすい文字は英語で記載する）

## commitのルール

### commitの分割

- 1つのcommitは1つの論理的な変更単位とする
- レビュワーが「このコミットは何をしているのか」を容易に理解できる単位にすること

### commitメッセージのprefix

commitメッセージには適切なprefixを付与すること。

prefix一覧:

| prefix | 用途 | 例 |
| -------------- | ------ | ----- |
| `feat:` | 新機能の追加 | `feat: ユーザー認証機能を追加` |
| `fix:` | バグ修正 | `fix: ログイン時のエラーハンドリングを修正` |
| `refactor:` | 機能変更を伴わないコード改善 | `refactor: 認証ロジックを関数に分離` |
| `style:` | フォーマット変更（動作に影響なし） | `style: インデントを統一` |
| `docs:` | ドキュメントのみの変更 | `docs: README にセットアップ手順を追加` |
| `test:` | テストの追加・修正 | `test: 認証機能のユニットテストを追加` |
| `chore:` | ビルド・ツール関連 | `chore: ESLint設定を更新` |

## PRのルール

### PRテンプレートの使用の徹底

テンプレートが存在する場合は必ず使用する。

### PRの分割

- 1つのPRは1つのレビュー可能な単位とする
- レビュワーが30分〜1時間程度で理解・レビューできるサイズを目安にすること

### PRタイトルのprefix

PRタイトルにはcommitと同様のprefixを付与すること。

prefix一覧:

| prefix | 用途 | 例 |
| -------------- | ------ | ----- |
| `feat:` | 新機能の追加 | `feat: ユーザー認証機能を追加` |
| `fix:` | バグ修正 | `fix: ログイン時のエラーハンドリングを修正` |
| `refactor:` | 機能変更を伴わないコード改善 | `refactor: 認証ロジックを関数に分離` |
| `style:` | フォーマット変更（動作に影響なし） | `style: インデントを統一` |
| `docs:` | ドキュメントのみの変更 | `docs: README にセットアップ手順を追加` |
| `test:` | テストの追加・修正 | `test: 認証機能のユニットテストを追加` |
| `chore:` | ビルド・ツール関連 | `chore: ESLint設定を更新` |

### PR説明文の必須要素

以下の3点を必ず記載すること。
テンプレートにそれらの要素が含まれている場合は、テンプレートに従うこと。

1. **概要（What）**: このPRで何が変わるか
2. **背景（Why）**: なぜこの変更が必要か
3. **確認方法（How to test）**: レビュワーがどのように動作確認できるか

### PR説明文の記述

- PRの説明には誇張表現や主観的な評価を含めない
- 文章として記述することはさけ、言い切り形式で簡潔に記載すること
